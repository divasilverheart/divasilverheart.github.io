Program Reflection & Portfolio Introduction

Program Reflection

The CS program gave me a lot of chances to revisit the same concepts from different angles, and by the time I got to the capstone most of them had started to feel automatic. CS 250 covered Agile and Scrum, which changed how I think about scope and planning. CS 320 was about writing code other people can test, which is a different problem than writing code that runs. CS 465 put both together in a full-stack MEAN application where every layer had to integrate correctly and the design decisions compounded on each other. Those courses together cover collaboration, stakeholder communication, and the kind of system thinking that makes software actually maintainable.

The more technical work built up the same way. CS 260 is where data structures stopped being abstract and started being something I could use to evaluate tradeoffs. CS 340 is where I started thinking about the database as part of the architecture rather than just something in the back that stores things. Security ran through a lot of the program in smaller pieces before it became explicit, and by the time I was looking at credentials sitting in a notebook that was about to go on a public GitHub repository, the habit of thinking about it was already there.

Artifact Summary

This portfolio has two artifacts. The first is the CS-340 Grazioso Salvare animal shelter dashboard, a full-stack Python application with a MongoDB backend and a Dash frontend. It covers software design and engineering and databases. The second is the CS-370 deep Q-learning treasure hunt game, a reinforcement learning project where an agent learns to navigate a maze. It covers algorithms and data structures. Together they cover all three capstone categories across two different problem domains.

The dashboard enhancements are about structure and the data layer. The software design work refactored a single Jupyter notebook into a multi-file Python application with separated concerns, environment variable configuration, and proper error handling. The database work added indexes based on the actual query patterns of the application so the database can look up records directly instead of scanning the whole collection on every filter. The game enhancement is a more focused algorithmic decision: replacing a plain list replay buffer with a deque with a fixed maxlen, which makes oldest-episode removal O(1) instead of O(n) and bounds memory across thousands of training epochs. One artifact is about building a system that holds up. The other is about choosing the right tool for a specific problem.

The Artifact

The artifact I chose for the Software Design and Engineering category is my CS 340 Client/Server Development project. It is a full-stack animal shelter dashboard built for a fictional client called Grazioso Salvare. The application pulls animal records from a MongoDB database and displays them in an interactive web dashboard that lets the user filter animals by rescue type. There are buttons to filter the data based on rescue type, an interactive data table with filters by column and pagination, a pie chart of breed distribution, and a map displaying the location of the selected animal.

Artifact Justification

I chose this project because it covers full-stack development through database integration, a basic UI, and backend logic, making it a good artifact for demonstrating overall software design skills. The original version worked, but everything was packed into a single Jupyter notebook cell with hardcoded credentials and improper documentation.

The enhancement converts the project from a Jupyter notebook into a standard Python application split across four files: animal_shelter.py for database interaction, data.py for filter logic, layout.py for the UI structure, and app.py as the main entry point. I also moved credentials into environment variables, added error handling on the database connection, removed unnecessary extra queries, and replaced hardcoded logic.

Enhancement Reflection

When I originally submitted the CS 340 project I knew it worked, but I also knew it wasn't clean. Coming back to it now, I caught things I wouldn't have noticed before, like the delete method counting records before deleting them instead of using the count the database already returns, or the map silently breaking if the column order changed. Those are the kinds of issues that don't come up in basic testing but matter in a real codebase. The main challenge was moving away from Jupyter — the original notebook used JupyterDash which handles app initialization differently than standard Dash, and splitting everything across separate files meant being more careful about imports and how the database connection gets shared.

CS-370 Deep Q-Learning Treasure Hunt

A reinforcement learning application where an AI agent learns to navigate an 8×8 maze to find treasure using deep Q-learning. The agent builds experience over training epochs and uses a neural network to decide which moves maximize its expected reward.

Read Enhancement Narrative

The Artifact

The artifact I selected for this category is my CS 370 Current and Emerging Trends in Computer Science project. It's a reinforcement learning application where an AI agent, represented as a pirate, learns to navigate an 8×8 maze to find treasure using deep Q-learning. The project contains TreasureMaze.py to define the maze components, GameExperience.py which handles the agent's memory and training data, and TreasureHuntGame.ipynb which is the main notebook where the neural network is built and the training loop runs.

Artifact Justification

This project gave me a good opportunity to look at data structures and algorithm efficiency in a machine learning context. The original version had a few real issues worth fixing. The memory system was using a plain Python list and manually deleting from the front every time it filled up, which is an O(n) operation that gets slower as memory grows. There was also a broken helper function that was never called, duplicate direction constants defined in two places, a logic bug in the movement method where one branch used if instead of elif, and leftover comments that didn't explain anything about the code.

The biggest change was replacing the list in GameExperience with a deque with a fixed maxlen, which handles oldest-episode removal automatically in constant time. I also fixed the if/elif bug in update_state, removed the broken and duplicate code, added descriptive variable names, and replaced comments with ones that accurately explain what each step is doing.

Enhancement Reflection

I was excited to approach this enhancement because I could only make performance improvements in one file when I originally worked on it. At the time I didn't think about what it would cost to delete from the front of a list on every step across thousands of training epochs. Coming back to it and making that change felt like a good example of the difference between code that runs and code that's well designed. The challenge was being careful not to break anything while making changes across three files — after removing the direction constants from the notebook, I had to make sure they were imported properly instead of just deleted.

CS-340 Animal Shelter Dashboard — Database Enhancement

A second pass on the CS-340 dashboard focusing specifically on the MongoDB layer: index optimization for the fields the dashboard queries most, and role-based access control so the data tier enforces permissions directly.

Enhanced — indexes.py Enhanced — animal_shelter.py
Read Enhancement Narrative

The Artifact

The artifact for this category is the same CS-340 project from Category One. It's a Grazioso Salvare animal shelter dashboard built with Python, MongoDB, and Dash. I'm using it again here because the database layer is its own area of focus separate from the software design work I did in the first enhancement.

Artifact Justification

The original project had hardcoded credentials sitting in the notebook and no indexes on the collection, meaning every filter the user clicked triggered a full scan of every document in the database. The credential issue was fixed as part of Category One when I moved everything to environment variables, so this enhancement focuses on the performance side. I wrote a setup script that creates a compound index on the four fields the dashboard filters query against, and a separate index on the record number field that gets hit on every insert. With those in place, the database can look up matching documents directly instead of scanning the whole collection every time.

Enhancement Reflection

The credential issue was the more eye-opening part of this. When I originally submitted the CS-340 project it was running on a closed course environment, so it never felt like a problem. Looking at it again, knowing it was about to go on a public GitHub repository made it obvious why that's not a habit that carries over to real work.

The indexing required thinking through how the application queries the database rather than just adding indexes and hoping for the best. The four filter options all query the same fields, so a compound index covering all of them made more sense than four separate ones. The record number index was a smaller thing, but that field gets queried on every single insert, so leaving it unindexed on a large collection would add up.
